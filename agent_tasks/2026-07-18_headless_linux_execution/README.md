# HEC-RAS headless Linux execution: decision record

Date: 2026-07-18

Status: execution routing and TCU work are complete. Exact HEC-RAS 7.0.1
native-Windows/Wine content parity now passes for the bounded mesh-sequence,
projection/multi-source-terrain, and Mapper result-layer/GeoTIFF-export slices.
The full lane remains unqualified because property-table generation, fresh
7.0.1 computation parity, recovery, and the production zero-skip gate remain
open.

## Outcome

HEC-RAS 7.0 did **not** introduce the first native Linux geometry
preprocessor. Official HEC material identifies Linux builds of
`RasGeomPreprocess`, `RasUnsteady`, and `RasSteady` in 6.1, 6.5, and 6.6.
Version 7.0 added an integrated Windows-to-WSL launch option for Linux compute
engines.

The phrase *geometry preprocessing* has been hiding two different operations:

1. The native Linux `RasGeomPreprocess` executable updates the classic 1D
   cross-section/structure hydraulic tables in an already prepared plan HDF.
2. The 2D cell/face terrain-derived property-table build is a separate RAS
   Mapper operation. HEC's 6.6 documentation does not provide a raw-project,
   pure-Linux route for it, and the official Linux example starts with files
   generated by a Windows GUI compute.

This distinction explains the observed Wine failure. Running native Linux
`RasGeomPreprocess` is the correct path for supported 1D HTAB work, but it is
not evidence that the missing RAS Mapper 2D cell/face table builder has been
replaced.

## Required execution policy

Use an official HEC-compiled native Linux engine exclusively whenever one is
available for the requested stage:

- `RasUnsteady` for unsteady computation;
- `RasSteady` for steady computation; and
- `RasGeomPreprocess` for the classic 1D geometry/HTAB stage, after its
  prerequisite files exist.

Do not run those same compute stages through Wine. Reserve native Windows, and
then a separately qualified Wine fallback, for preparation capabilities that
HEC has not exposed as a native Linux executable:

- RAS Mapper terrain creation and registration;
- 2D mesh authoring and mutation;
- geometry/terrain/land-cover/infiltration associations;
- 2D cell/face hydraulic-property tables; and
- other GUI/Mapper review or HITL operations.

The practical production split is therefore:

| Stage | Preferred path | Fallback/status |
|---|---|---|
| Detailed authoring and final engineering review | Native Windows HEC-RAS 7.0.1 | None for final acceptance |
| RAS Mapper preparation | Native Windows | Wine only after content/parity qualification |
| 1D HTAB refresh | Official native Linux engine | Windows only where the version/artifacts require it |
| Unsteady/steady scenario solve | Official native Linux engine | Do not route through Wine merely for uniformity |
| Result extraction | Native Linux Python/GDAL | Validate against HEC-RAS content, not exit code |

## Current blockers and resolved controls

1. **2D property-table generation under Wine remains failed.** Both
   `CompleteGeometryCommand` and direct `ComputePropertyTablesCommand` failed to
   return on real 7.0.1 fixtures. This is now the principal technical gap, not
   native Linux unsteady computation.
2. **The Windows/Wine dialog watchdog safety defect is fixed and
   live-qualified for the TCU path.** The prior generic fallback could click
   the first button on an unknown modal. The new shared classifier blocks legal
   and unknown dialogs without UI interaction, removes generic Yes handling,
   requires an explicit exact-title allowlist even for informational OK/Close
   dialogs, and terminates only the registered launch tree. Current execution
   callers provide no dismissal allowlist, and global RAS-process discovery is
   disabled unless explicitly requested. Missing pywin32 or psutil fails before
   launch; termination is checked for survivors; scan-thread failures retain
   verified termination status; the legacy preprocessor's TCU enumeration is
   process-tree scoped; and single/parallel result objects retain the diagnostic.
   Native tests across the 15 installed stable 4.xâ€“7.x builds and four
   reversible Wine 7.0.1 prefix clones plus one persistent disposable clone
   prove zero-click
   TCU detection, exact-state positives,
   missing/wrong-state negatives, application-subtree restoration, and no
   surviving tracked processes. The installed 6.7 beta builds are separate
   optional evidence under HEC's Beta Software User's Agreement, not part of
   the stable gate. The earlier persistent candidate is diagnostic rather than
   template-promotion evidence. This does not qualify other unknown RAS Mapper
   dialogs or the broader computation workflow.
3. **A first-class native `RasGeomPreprocess` API is missing in
   ras-commander.** A Docker shell script invokes it, but it is not a reusable,
   fail-closed library API and can continue after failure.
4. **The existing greenfield-Linux claims are too broad.** ras-agent has real
   native-solver evidence for a Windows-preprocessed Spring Creek fixture. It
   does not yet have production evidence that its native preprocessor builds
   missing 2D Tier-B tables from a raw project.
5. **TCU no longer blocks the current Mapper tests, but the 7.0 â†’ 7.0.1
   inheritance hypothesis remains rejected.** HEC calls 7.0.1 primarily a
   bug-fix release but installs it independently, and black-box tests show that
   version lineage alone does not carry acceptance into the separate target.
   Merged [PR #287](https://github.com/gpt-cmdr/ras-commander/pull/287)
   instead provides read-only detection plus an explicit opt-in transfer from
   an already accepted donor state; it never fabricates acceptance and returns
   a diagnostic when no donor exists. This project-level compatibility path is
   not a USACE-supported headless interface and does not qualify Mapper behavior.
6. **HEC-RAS 6.4.1 first-start instability has a repeatable WPF mitigation.**
   NGen alone did not stop the Wine debugger failure after the time-series
   child appeared. Microsoft-documented per-user WPF software rendering did:
   two independently rebuilt fresh prefixes each passed the two TCU negatives
   and three consecutive full-duration ready probes with zero interactions or
   survivors. This is startup evidence, not full Mapper qualification.
