# Official Linux support matrix

This matrix deliberately distinguishes an official HEC-compiled engine from a
full vendor-supported Linux desktop application. HEC uses language such as
*built and tested*, *test setup*, and *compute engines only*. The evidence does
not justify describing the classic HEC-RAS GUI or RAS Mapper as a supported
native Linux product.

Repository artifact-inspection code expects the 7.0.1 special installer to
contain `RasGeomPreprocess`, `RasUnsteady`, and `RasSteady`. That is useful
artifact evidence, but the installed official package should be fingerprinted
before those exact per-executable claims are placed in public documentation.

## What the 6.5/6.6 Linux workflow actually starts with

The official 6.5 and 6.6 examples begin with:

- a `.b##` unsteady input file;
- an `.x##` file described as 1D geometry information; and
- a `.p##.tmp.hdf` derived from a Windows GUI plan HDF with the Results group
  removed.

The notes say to run `RasGeomPreprocess` when the geometry input has changed
outside the GUI. The executable updates hydraulic tables in that prepared plan
HDF. This is materially different from compiling a new 2D model from geometry
text, terrain, land cover, and mesh controls.

The 2D manual separately states that each cell and face is preprocessed into
terrain-derived property curves through RAS Mapper. If the user does not run
that operation in Mapper, the Windows HEC-RAS UI starts it as a separate pre-run
process before the existing 1D preprocessor. No located 6.6 or 7.0 document
names a standalone native Linux equivalent for this RAS Mapper operation.

## Correct reading of the 7.0 changelog

The 7.0 new-feature entry says that Windows can launch Linux-compiled compute
engines using WSL when the special package is installed. It describes this as a
testing/cloud Monte Carlo preparation feature and notes WSL startup overhead.
It does not say that `RasGeomPreprocess` first appeared in 7.0, nor does it
announce a standalone native 2D RAS Mapper table builder.

Version 7.0.1 is primarily a 7.0 bug-fix release. Pinning it is this project's
qualification policy, not an HEC recommendation, and it should not be credited
with an unverified new geometry-preprocessing capability. HEC also documents
7.0.1 as an independent installation. Black-box TCU controls show that 7.0
acceptance state does not start 7.0.1, so the bug-fix designation is not
technical acceptance-state inheritance.

## TCU evidence boundary

HEC's published terms state that use of HEC software constitutes voluntary
acceptance. They grant rights to install, use, copy, and distribute the
software subject to recipient agreement. No official source located here
describes:

- the first-run dialog's storage mechanism;
- registry/config portability;
- cloning an accepted VM or Wine prefix; or
- transferring acceptance state between native Windows and Wine.

HEC also prohibits modification, decompilation, disassembly, unobfuscation, and
reverse engineering of HEC Software. Same-version state provisioning is
therefore an undocumented internal experiment, not a USACE-endorsed technical
procedure; recorded user authority does not make it vendor-supported or remove
the need for compliance review. The supported fallback is a user-visible
exact-version acceptance flow. The stable qualification scope is the 15 exact
installed 4.x–7.x builds recorded in `tcu_acceptance_state.md`; it is not every
historical point release. The installed 6.7 beta builds are outside that stable
scope and additionally fall under HEC's separate Beta Software User's
Agreement.
